Honest Validator Self-Equivocation: SkipVote + NotarizeVote for the Same Slot

Repo: ton-blockchain/ton Branch: testnet Commit: 3bb6abc Scope: Primary — validator/consensus/simplex/ Date: 2026-03-14

Summary

An honest Simplex validator can broadcast both a SkipVote and a NotarizeVote for the same slot, producing a self-equivocation from a validator that is behaving correctly and has not been compromised. This violates the core Simplex safety invariant that a validator votes at most once per slot. The bug arises from an incomplete guard in three locations across two code paths:

  • Code Path A — alarm() race: alarm() and try_notarize() use disjoint guards, allowing both to broadcast for the same slot concurrently.
  • Code Path B — start_up() restart: On every restart, a validator broadcasts SkipVote for its active window slots without checking whether it already cast NotarizeVote for those slots before the crash.

Both paths are dynamically confirmed via the test-consensus harness.

Impact

Code Path A — alarm() Race (consensus.cpp)

Root Cause

alarm() and try_notarize() share state on a slot but use disjoint boolean guards. alarm() checks only !voted_final before broadcasting SkipVote. try_notarize() sets voted_notar and broadcasts NotarizeVote after resuming from co_await ValidationRequest — without checking whether a SkipVote was sent during the suspension.

validator/consensus/simplex/consensus.cpp

Bug A1 — alarm(): missing !voted_notar guard

// consensus.cpp ~line 144
td::actor::Task<> alarm() {
  for (size_t i = first_...; i < last_...; ++i) {
    auto slot = bus_->slots.get(i);
    if (slot && !slot->state->voted_final) {       // BUG: must also check !voted_notar && !voted_skip
      owning_bus().publish<BroadcastVote>(SkipVote{i});
    }
  }
}

Bug A2 — try_notarize(): missing !voted_skip guard

// consensus.cpp ~line 228
td::actor::Task<> try_notarize(size_t slot_idx, ...) {
  auto res = co_await ValidationRequest{candidate};
  // alarm() may have fired here and broadcast SkipVote — not checked
  slot->state->voted_notar = candidate->id;
  owning_bus().publish<BroadcastVote>(NotarizeVote{...});  // BUG: should check !voted_skip first
}

Bug A3 — check_invariants(): missing detection case

// pool.cpp ~line 209
void check_invariants() {
  if (finalize_.has_value() && skip_.has_value()) { ... report ... }
  if (finalize_.has_value() && notarize_.has_value()) { ... report ... }
  // MISSING: notarize_ + skip_ combination not detected:
  // if (notarize_.has_value() && skip_.has_value()) { report misbehavior; }
}

Code Path B — start_up() Restart Equivocation (consensus.cpp)

Root Cause

On restart, start_up() loads bootstrap_votes from the persistent DB (including any NotarizeVote cast before the crash), then immediately broadcasts SkipVote for all slots in the current first_nonannounced_window range using only a !voted_final guard. Peers who received the pre-crash NotarizeVote now also receive a SkipVote from the same validator for the same slot — equivocation.

validator/consensus/simplex/consensus.cpp
// consensus.cpp ~line 82
void start_up() {
  ...
  if (auto window = bus->first_nonannounced_window) {
    for (size_t i = *window; i < last_slot; ++i) {
      auto slot = bus->slots.get(i);
      if (slot.has_value() && !slot->state->voted_final) {   // BUG: must also check !voted_notar
        owning_bus().publish<BroadcastVote>(SkipVote{i});
      }
    }
  }
}

Why the DB persists: The test-consensus harness uses a TestDbImpl backed by a shared_ptr<DbInner> (test-consensus.cpp:610, 689). The DbInner is preserved across gremlin kills, so bootstrap_votes correctly reflects pre-crash state on restart.

✅ Dynamic Evidence (Path B):
Gremlin (--gremlin-kills-leader, period 2–4s, downtime 1–2s): 20 leader restarts observed, 19 showing immediate post-restart SkipVote drops from the restarted validator.
Example: Validator 0 restarted at 15:55:24.007 → SkipVote{slot=5} at 15:55:24.060 (53ms later) → pool.cpp:525: slot already finalised.
